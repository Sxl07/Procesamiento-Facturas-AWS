# Technical Decisions

## Context
This document records the main technical decisions defined before Sprint 1 for the Invoice AI Platform project.
The objective is to keep architecture, responsibilities, and security boundaries clear from the start.

## 1. Frontend deployment
Angular will be deployed using Amazon S3 for static hosting and Amazon CloudFront as the CDN and edge distribution layer.
This keeps the frontend simple, scalable, and aligned with AWS-native hosting patterns for SPA applications.

## 2. Backend responsibility
Spring Boot will own the core business logic of the platform.
The backend will orchestrate invoice workflows, validation rules, persistence logic, signed URL generation, and integrations with AWS services.

## 3. File upload model
PDF uploads will use pre-signed S3 URLs generated by the backend.
The frontend must never contain AWS credentials or direct privileged access to AWS resources.

## 4. Operational data source
DynamoDB will be the operational source of truth for invoice processing data.
It will store metadata, processing states, references to stored documents, extracted fields, and workflow status.

## 5. Search boundary
OpenSearch will be used only for search and retrieval use cases.
It will not be treated as the primary operational database.
DynamoDB remains the source of truth.

## 6. Invoice extraction engine
Amazon Textract AnalyzeExpense will be used for invoice data extraction.
This service is specialized for invoice-like financial documents and fits the project requirements better than a generic OCR-only flow.

## 7. Chatbot architecture
Amazon Lex will be used for the chatbot interface.
Lambda fulfillment will implement intent handling and backend integration logic.

## 8. IAM strategy
IAM access will follow separated roles and least privilege principles.
Each major component (backend, Lambda functions, CI/CD, and support services) must use dedicated roles or policies limited to the actions and resources it actually needs.

## 9. Infrastructure as code
Critical AWS resources must be defined through AWS CDK and versioned in the repository.
Manual console creation should be avoided for persistent or security-relevant infrastructure.

## 10. Security baseline
The project starts with the following security baseline:
- S3 Block Public Access enabled
- Encryption enabled by default for stored data
- CloudTrail enabled
- Budget monitoring enabled
- No secrets stored in source code or application.yml
- Sensitive configuration should use AWS Systems Manager Parameter Store or AWS Secrets Manager

## 11. Environment strategy
The infrastructure is organized by environment (at minimum dev and prod).
Resource naming, tagging, and deployment configuration must remain environment-aware from the beginning.

## 12. Monorepo structure
The project uses a monorepo with separated folders for infrastructure, frontend, backend, and Lambda functions.
This keeps related code versioned together while preserving clear boundaries between responsibilities.
